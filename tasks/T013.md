# T013: Distributed Worker Core & Monitoring Execution

## 基本信息

- 所属需求: Phase 3.1: Job Scheduler (Global)
- 深度/宽度: L2 / M
- 负责人: Claude (gsd-planner)
- 前置: T012 📝 (Dispatcher)

---

## 输入

- `monitoringWorker` (BullMQ): 消费任务的 Worker。
- `AdapterFactory`: 用于根据 `aiPlatform` 和 `countryCode` 获取适配器。
- `LLMParser`: 用于解析模型原始输出。
- Job Data: `{ promptId, countryCode, aiPlatform, tenantId }`。

---

## 输出

- `src/lib/queue/worker.ts`: 更新 Worker 处理核心逻辑。
- `monitoring_results` 表 (DB): 任务执行结果存储。

---

## 验收标准（技术视角）

- [ ] **执行逻辑实现**:
  - [ ] 根据 `promptId` 从 DB 获取 Prompt Content。
  - [ ] 调用 `AdapterFactory.getAdapter(aiPlatform, countryCode)`。
  - [ ] 调用 `adapter.query(content)` 获取 `rawResponse`。
  - [ ] 调用 `LLMParser.parse(rawResponse)` 获取结构化数据。
  - [ ] 将 `rawResponse` + `structuredData` 存入 `monitoring_results` 表。
- [ ] **异常捕获机制**:
  - [ ] 如果适配器调用失败（例如 429 Rate Limit 或 503 Overloaded），应抛出 `UnrecoverableError` (BullMQ 语义) 或普通 Error 以触发重试。
  - [ ] 如果解析失败，应存储 `rawResponse` 并在 `structuredData` 中标注解析错误。
- [ ] **分布式运行安全**:
  - [ ] 确保 Worker 是无状态的，支持多副本并行运行。
  - [ ] 控制 `concurrency` (并发数)，防止触发下游 LLM API 的单账号 QPS 限制。
- [ ] **状态反馈**:
  - [ ] Job 完成后向 BullMQ 返回处理详情（耗时、结果 ID）。
